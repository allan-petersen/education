A relational database does not store data in a hierarchical way. Then how do I get the data in a hierarchical manner? Here we get to know about how to use the hierarchical querying feature which Oracle has given. This article talks about how you can interpret the hierarchical query conceptually and build hierarchical queries catering your needs.

Using hierarchical queries, you can retrieve records from a table by their natural relationship. Be it a family tree or a employee/manager tree or what ever.

Tree walking enables you to construct a hierarchical tree if the relationship lie in the same table. For instance, a manager column which exists in the emp table which defines the managerial hierarchy.

But if we have to walk down the tree and check who all are reporting to Jones, Blake and Clark (recursively)

SELECT empno, 
       ename, 
       job, 
       mgr, 
       hiredate 
FROM   emp 
START WITH mgr IS NULL 
CONNECT BY PRIOR empno = mgr
We will quickly see what are all the key words used in this query.

START WITH – Specifies the root rows of the hierarchy or in other words, where to start parsing from. This clause is necessary for true hierarchical queries

CONNECT BY PRIOR – This explains the relationship between the parent and the child.

PRIOR – This is used to achieve the recursive condition (The actual walking)

Direction of walking the tree
To explain more on the CONNECT BY clause, this is used to determine if you are walking from top to bottom or bottom to top.

Pruning branches/children
There might be business requirements to partially retrieve a hierarchical tree and to prune branches.
If you do not want to do so, use the where condition to restrict the branch but process the child row

SELECT empno, 
       ename, 
       job, 
       mgr, 
       hiredate 
FROM   emp 
WHERE  ename <> 'JONES' 
START WITH mgr IS NULL 
CONNECT BY PRIOR empno = mgr
This will restrict the value Jones in the result set but will still will retrieve Scott and Ford.

Please refer to the attached Picture to get a complete understanding.

To Restrict the value clark and its children, you should be adding the condition after the CONNECT BY

SELECT empno, 
       ename, 
       job, 
       mgr, 
       hiredate 
FROM   emp 
START WITH mgr IS NULL 
CONNECT BY PRIOR empno = mgr 
                 AND ename <> 'CLARK'

START WITH manager_id IS NULL:
This clause identifies the root record(s) in the hierarchy (e.g., the top-level employees or executives).

CONNECT BY PRIOR employee_id = manager_id:
This clause defines the relationship between parent and child rows.

CONNECT_BY_ROOT employee_name:
Regardless of the depth in the hierarchy, this function returns the employee_name of the root row from which the hierarchical traversal began.				 

WHERE CONNECT_BY_ISLEAF = 1:
This filters the result set to only include rows that are leaf nodes (i.e., employees who do not have any subordinates).

Conditions
==========
Here’s an example query that demonstrates conditions for top nodes, intermediate nodes, and leaf nodes:

sql

SELECT employee_id,
       employee_name,
       LEVEL,
       CASE
         WHEN CONNECT_BY_ISLEAF = 1 THEN 'Leaf'
         WHEN LEVEL = 1 THEN 'Top'
         ELSE 'Intermediate'
       END AS node_type
FROM employees
START WITH manager_id IS NULL              -- Only rows with no manager are top nodes.
CONNECT BY PRIOR employee_id = manager_id
-- Optionally, you can add a WHERE clause to further filter:
-- WHERE (For top nodes: conditions in START WITH,
--        For leaf nodes: CONNECT_BY_ISLEAF = 1,
--        For intermediate: LEVEL > 1 AND CONNECT_BY_ISLEAF = 0)
;

Condtions on all
================
Below are a couple of approaches to force a condition to apply to all nodes (top, intermediate, or leaf) in an Oracle hierarchical query.

1. Pre-filtering in a Subquery
One common method is to filter the table data first (in an inline view) before applying the hierarchical query. This way the condition is enforced on every row before the hierarchy is built.

For example, if you need every node to have status = 'ACTIVE', you can write:

SELECT employee_id,
       employee_name,
       LEVEL
FROM (
  SELECT *
  FROM employees
  WHERE status = 'ACTIVE'  -- condition applies to all rows
)
START WITH manager_id IS NULL
CONNECT BY PRIOR employee_id = manager_id;
In this approach, every row considered by the hierarchical query already meets the status condition.

2. Including the Condition in the CONNECT BY Clause
Another approach is to incorporate the condition directly into your CONNECT BY clause (by adding an AND condition). This forces every parent–child relationship to honor the condition. For instance:


SELECT employee_id,
       employee_name,
       LEVEL
FROM employees
START WITH manager_id IS NULL
CONNECT BY PRIOR employee_id = manager_id AND status = 'ACTIVE';
This ensures that a child row is only linked to its parent if the child also meets the status condition. Note that if a row fails the condition, its branch will be pruned from the hierarchy.

Additional Considerations
WHERE Clause on the Final Result:
While you could also put a condition in the final WHERE clause, remember that the WHERE clause is applied after the hierarchy is built. This means that if a row (or even an entire branch) does not meet the condition, it will be filtered out from the final result—but it might have already affected the hierarchical structure (and pseudo columns like LEVEL).


Choice of Method:
If the condition should determine the structure (i.e., which rows get linked), it’s often better to include it in the CONNECT BY clause or pre-filter using a subquery. 
If it’s solely for the final output, then placing it in the WHERE clause might be acceptable.


By pre-filtering the rows or integrating the condition into the CONNECT BY clause, you ensure that your condition (for example, status = 'ACTIVE') applies consistently to all nodes—top nodes, intermediate nodes, and leaf nodes—in your hierarchical query.



sys_connect_by_path
===================
Below is a breakdown of what the expression does:

sys_connect_by_path(sor.eanlocation, '|'):
This function is used in Oracle hierarchical queries. It concatenates the values of the column sor.eanlocation along the hierarchy using the pipe (|) as a delimiter. For example, if the hierarchy has values A, B, C, then this expression would return something like:
│A│B│C
(with a leading pipe)


ltrim(sys_connect_by_path(sor.eanlocation, '|'), '|'):
This removes the leading pipe character from the string. So from │A│B│C you would get:
A│B│C


ltrim(sys_connect_by_path(sor.eanlocation, '|') || '|', '|'):
This part appends an extra pipe to the string before performing further operations. In our example, it would result in:
A│B│C|
This ensures that even if sor.eanlocation is the only element (i.e., no pipes occur later), there will be a delimiter to find.


instr(ltrim(sys_connect_by_path(sor.eanlocation, '|') || '|', '|'), '|', 1):
The INSTR function is used here to locate the position of the first occurrence of the pipe (|) in the string (after left-trimming). Given our example "A│B│C|", this returns the position of the first pipe after "A". This position tells us where the first token ends.
